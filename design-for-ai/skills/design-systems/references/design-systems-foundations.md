# Design systems foundations — tokens, atoms, and composition

The constructive layer: how to build a design system on top of a locked DESIGN.md. This reference covers the semantic token tier hierarchy, atomic design composition, the W3C DTCG token format, and Kholmatova's functional vs perceptual pattern model.

**Sources:** Frost *Atomic Design* (2013/2016); Kholmatova *Design Systems* (2017, Smashing); W3C Design Tokens Community Group (DTCG) format (stable Oct 2025); Perez-Cruz *Expressive Design Systems* (2022); W3C Design Tokens format specification.

**Prerequisite:** A locked DESIGN.md with global tokens generated by `palette.mjs`. This reference builds the semantic tiers on top of those globals; it does not replace or re-derive them.

## KEY DEFINITIONS

> **Design token**: a named, platform-agnostic design decision, expressed as a key-value pair. Tokens are the single source of truth for design decisions across platforms (web, iOS, Android, print).

> **Global token**: a raw design decision — a specific value in the system's vocabulary. `color-blue-500: #2563eb`. Generated by `palette.mjs`; not for direct component consumption.

> **Alias / semantic token**: a named role that maps to a global value. `color-text-primary: {color-blue-500}`. Encodes intent ("primary text uses blue-500") without coupling consumers to the raw value.

> **Component token**: a scoped token for a specific component. `button-background: {color-text-primary}`. Lets one component be restyled without changing the alias tier.

> **Atom (Frost)**: the smallest non-decomposable UI unit — a button, an input field, a label, a color swatch, a font sample.

> **Molecule (Frost)**: a functional combination of atoms — a search field (input + button + label).

> **Organism (Frost)**: a complex UI section built from molecules and atoms — a site header (logo + nav + search).

> **Template (Frost)**: the page-level arrangement of organisms, with placeholder content. Shows structure and layout without real content.

> **Page (Frost)**: a specific instance of a template with real content — where design meets reality and edge cases surface.

> **Functional pattern (Kholmatova)**: a shared UI behavior — navigation shell, form layout, data table. The structural grid.

> **Perceptual pattern (Kholmatova)**: a shared visual quality — color use, spacing rhythm, typographic hierarchy. How the system feels.

## THE THREE-TIER TOKEN HIERARCHY

Design systems work because they separate three kinds of decisions: what values exist (global), what role a value plays (alias), and what a specific component uses (component). Conflating these tiers produces a system that's hard to maintain and impossible to theme.

### Tier 1: Global tokens (from palette.mjs)

The vocabulary. Raw values, no semantic meaning. Generated by the core `design` skill's `palette.mjs` and locked in DESIGN.md.

```json
{
  "color-blue-100": { "$value": "#dbeafe", "$type": "color" },
  "color-blue-500": { "$value": "#2563eb", "$type": "color" },
  "color-blue-900": { "$value": "#1e3a8a", "$type": "color" },
  "space-1": { "$value": "0.25rem", "$type": "dimension" },
  "space-4": { "$value": "1rem", "$type": "dimension" },
  "font-size-base": { "$value": "1rem", "$type": "fontSizes" }
}
```

Rule: global tokens are never consumed directly by components. They are referenced by alias tokens only.

### Tier 2: Alias / semantic tokens

The design decisions. Maps global values to semantic roles.

```json
{
  "color-text-primary": { "$value": "{color-blue-900}", "$type": "color", "$description": "Primary text color — body copy and headings" },
  "color-text-secondary": { "$value": "{color-blue-500}", "$type": "color", "$description": "Secondary/supporting text" },
  "color-surface-default": { "$value": "{color-blue-100}", "$type": "color", "$description": "Default surface background" },
  "color-interactive-default": { "$value": "{color-blue-500}", "$type": "color", "$description": "Default interactive element color" },
  "space-content-gap": { "$value": "{space-4}", "$type": "dimension", "$description": "Standard gap between content sections" }
}
```

Rule: alias tokens encode intent ("primary text is blue-900"), not implementation. A color rename (blue-900 → slate-900) changes only global tokens and DESIGN.md — the alias tier is unaffected.

### Tier 3: Component tokens

Scoped to a single component. References alias tokens, not globals.

```json
{
  "button-background": { "$value": "{color-interactive-default}", "$type": "color" },
  "button-text": { "$value": "#ffffff", "$type": "color" },
  "button-padding-x": { "$value": "{space-4}", "$type": "dimension" },
  "card-background": { "$value": "{color-surface-default}", "$type": "color" },
  "card-padding": { "$value": "{space-content-gap}", "$type": "dimension" }
}
```

Rule: component tokens isolate the component from alias changes. Restyling a button changes only its component tokens, not the alias tier. This is where component-level theming (dark mode variant, size variant) happens.

### When to use each tier

| Decision | Tier |
|----------|------|
| What colors exist in the system | Global (palette.mjs) |
| What role a color plays (primary text, interactive, surface) | Alias |
| What value a specific component uses | Component |
| Multi-brand theming | Swap global tier; alias + component tiers unchanged |

## W3C DTCG TOKEN FORMAT

**Source:** W3C Design Tokens Community Group specification (stable October 2025).

The DTCG format is the emerging standard for platform-agnostic token delivery. Style Dictionary and Cobalt support it natively.

### Token format

Each token is an object with reserved `$`-prefixed keys:

```json
{
  "token-name": {
    "$value": "<value>",
    "$type": "<type>",
    "$description": "<human-readable description>",
    "$extensions": {}
  }
}
```

### Token types

| `$type` | Values |
|---------|--------|
| `color` | Hex, RGB, HSL, OKLCH |
| `dimension` | px, rem, em, % |
| `fontFamily` | Font family name or stack |
| `fontWeight` | Numeric (400, 700) or keyword |
| `fontSize` | rem, px |
| `lineHeight` | Unitless ratio or px |
| `letterSpacing` | em, px |
| `duration` | ms (for motion tokens) |
| `cubicBezier` | Easing curve [x1, y1, x2, y2] |
| `shadow` | Shadow value object |

### Groups and dotted paths

Tokens are organized in nested groups using dotted path notation:

```json
{
  "color": {
    "text": {
      "primary": { "$value": "{color.blue.900}", "$type": "color" },
      "secondary": { "$value": "{color.blue.500}", "$type": "color" }
    },
    "surface": {
      "default": { "$value": "{color.blue.100}", "$type": "color" }
    }
  }
}
```

Resolved path: `color.text.primary`. Style Dictionary transforms this to platform-specific output (`--color-text-primary` in CSS, `ColorTextPrimary` in Swift, etc.).

### References (aliases)

Use curly-brace notation: `"{color.blue.900}"`. References must resolve — a reference to a non-existent token is a validation error. Style Dictionary validates references at build time.

## ATOMIC DESIGN — FROST'S COMPOSITION MODEL

**Source:** Frost *Atomic Design* (blog series 2013; book 2016).

Atomic design is a **composition model** — a way of thinking about how UI components relate to each other. It is not a file structure, not a Figma page organization, and not a one-to-one mapping to a component library's public API.

### The five levels

**1. Atoms**
The smallest non-decomposable units. Cannot be broken down without losing their function.
- HTML elements: `<input>`, `<button>`, `<label>`, `<a>`, `<img>`
- Design tokens used in isolation: a color swatch, a type sample, a spacing unit
- Icon, avatar, badge, spinner

Atoms define the visual grammar of the system — they carry tokens directly and express the perceptual patterns.

**2. Molecules**
Functional combinations of atoms that perform a specific UI task.
- Search field: `<input>` + `<button>` + `<label>`
- Form field: `<input>` + `<label>` + error message atom
- Card header: avatar atom + heading atom + badge atom

The guiding principle (Frost): molecules should do *one thing well*. A molecule that does three things is an organism.

**3. Organisms**
Complex, self-contained UI sections composed of molecules and atoms.
- Site header: logo atom + nav molecule + search molecule + user-menu molecule
- Product card grid: multiple product-card molecules + pagination molecule
- Data table: header organism + row molecules + pagination molecule

Organisms represent distinct sections of an interface. They can be extracted and placed in different contexts without losing meaning.

**4. Templates**
Page-level wireframes showing how organisms arrange on a page, with placeholder content. The structural skeleton before real content.
- A landing page template shows hero organism, social proof organism, and CTA sections in order.
- A dashboard template shows nav organism, stat cards, and data table in the main content area.

Templates are design artifacts, not code artifacts. They answer the question: "How do these sections relate spatially?"

**5. Pages**
Specific instances of templates with real content. Where abstract design meets concrete reality — where edge cases (too-long titles, missing images, error states) surface.

Pages reveal whether the template works with real data, real copy, and real constraints.

### Applying atomic design in practice

- Start by **auditing existing UI** for atoms (what are the smallest repeating elements?) before designing new atoms.
- The composition direction is bottom-up: atoms define the vocabulary; molecules and organisms define the grammar; templates and pages define the meaning.
- Atomic design is **technology-agnostic**: applies to React components, Vue components, CSS utility classes, or Figma component libraries.
- Frost's own note: the levels exist on a spectrum. The goal is consistent, reusable components — not a bureaucratic level-classification exercise.

## FUNCTIONAL VS PERCEPTUAL PATTERNS — KHOLMATOVA

**Source:** Kholmatova *Design Systems* (2017, Smashing Magazine), Chapter 3.

Two kinds of patterns make up a design system. Keeping them distinct prevents a common failure: a system that has consistent structure but inconsistent feel (or vice versa).

### Functional patterns

> Shared UI behaviors and structures. The *what* a component does.

Examples: the navigation shell, the form layout, the modal dialog, the data table, the notification system, the onboarding flow. Functional patterns encode interaction conventions — they are the structural grid of the system.

Functional patterns are closest to Frost's molecules and organisms. They should be documented with: the interaction states (default, hover, focus, loading, error, empty), the accessibility requirements (ARIA roles, keyboard navigation), and the usage guidelines (when to use this pattern vs. an alternative).

### Perceptual patterns

> Shared visual qualities. The *how* a component feels.

Examples: color use (interactive elements are always blue-500, destructive actions are always red-600), spacing rhythm (content sections are separated by `space-content-gap`), typographic hierarchy (H1 is always the primary heading style, body copy is always the body style), motion language (entries ease out, exits ease in).

Perceptual patterns are closest to Frost's atoms and the token system. They are enforced by the token tiers — consistent perceptual patterns emerge from consistent token use, not from per-component style decisions.

### Where inconsistency lives

Inconsistency in a design system almost always falls into one of two categories:
1. **Inconsistent functional patterns**: two flows that do the same job in different ways (two modal patterns, two date picker patterns, two navigation treatments).
2. **Inconsistent perceptual patterns**: correct functional patterns applied with inconsistent visual style (same button structure, different border-radius or padding across instances).

Diagnose with this lens: is the inconsistency functional (structural) or perceptual (stylistic)? The fix is different in each case — consolidate the pattern (functional) or enforce the token (perceptual).

## COMPONENT COMPOSITION PRINCIPLES

Principles for building components that stay composable and maintainable as the system scales.

### Single responsibility

Each component has one job. A component that handles layout, data fetching, and interaction state is three components. (Mirrors Frost's molecule principle: one thing well.)

### Inversion of control (slot/children pattern)

Components accept content as children or via slots rather than encoding all variants as props. A `Card` component that accepts `<CardHeader>`, `<CardBody>`, and `<CardFooter>` as children is more composable than a `Card` with `title`, `body`, and `footer` props — the slot model allows any atom or molecule inside any slot.

### Token consumption, not hard-coded values

Components consume component tokens (`button-background: {color-interactive-default}`), never raw hex values. Hard-coded values break theming and multi-brand support.

### State completeness

Every component documents and implements all 8 interaction states: default, hover, focus, active, disabled, loading, error, success. Incomplete state coverage is the most common component quality gap.

### Accessibility contract

Every component ships with: keyboard navigation defined, ARIA roles named, focus management specified, color-contrast requirement documented (from the token tier — if the alias token meets AA contrast, the component inherits it; if the component adds elements, they must also meet AA).

## MULTI-BRAND THEMING

The token tier hierarchy makes multi-brand theming a global-tier swap.

### How it works

1. **Component and alias tiers are shared** across brands. `button-background: {color-interactive-default}` is the same for all brands.
2. **Alias tokens are shared** across brands. `color-interactive-default: {color-blue-500}` is the same for all brands.
3. **Global tokens differ per brand**. Brand A: `color-blue-500: #2563eb`. Brand B: `color-blue-500: #7c3aed` (a purple). The alias tier maps `color-interactive-default` to `{color-blue-500}` in both brands — but the resolved hex is different.

### In practice

- Generate a DESIGN.md and run `palette.mjs` once per brand, producing distinct global token files.
- The alias and component tier files are shared (or thin wrappers that reference the same aliases).
- The build system (Style Dictionary, Cobalt) takes the brand-specific global file + the shared alias file + the shared component file and resolves the full token graph.
- Test the brand swap by rendering the same component library with both global token sets side-by-side.

### When multi-brand theming is worth it

Only when two or more products share an underlying component architecture but need distinct visual expressions. The overhead of the three-tier system pays off at scale (Mall: "Design That Scales" documents the ROI threshold). A single-product team doesn't need it — alias tokens are sufficient.

## DESIGN REVIEW CRITERIA

When auditing an existing design system or token implementation:

| Criterion | Pass condition | Fail signal |
|-----------|---------------|-------------|
| **Tier separation** | Components consume component tokens only; alias tier is not bypassed | Hard-coded hex in component styles; global tokens used directly in components |
| **Reference resolution** | All `{token.references}` resolve to existing tokens | Dangling references; build errors in Style Dictionary |
| **Semantic labeling** | Alias tokens encode intent (`color-text-primary`), not implementation (`blue-900`) | Alias names that are just renamed globals (`alias-blue-900`) |
| **State completeness** | All 8 states implemented per component | Missing disabled, loading, or error state |
| **Perceptual consistency** | Same visual property uses the same token everywhere | Button border-radius differs from card border-radius for no reason |
| **Functional consistency** | Same interaction pattern uses the same component | Two modal implementations; two date picker patterns |
| **Accessibility contract** | All components have ARIA, keyboard nav, and AA contrast documented | No ARIA roles; focus ring removed; contrast not verified |

## COMMON MISTAKES

| Mistake | What it produces | Fix |
|---------|-----------------|-----|
| Alias tokens that just rename globals | A two-tier system masquerading as three; theming still breaks | Map globals to semantic intent, not just new names |
| Hard-coded hex in components | Breaks theming and dark mode; every brand change requires component edits | All values flow through the token tiers |
| Atomic design as a file structure | Deeply nested folder hierarchies that become resentment artifacts | Use atomic design as a mental model; organize files by consumer, not level |
| Global tokens consumed in components | Same as hard-coding — the alias tier exists to abstract the global values | Alias tier is mandatory between global and component tiers |
| Style Dictionary pinned to a major version | Breaking API changes; DTCG format support varies by version | Reference current docs; document the version in use |
| Component tokens that reference other component tokens | Cross-component token dependencies produce cascade failures at theme time | Component tokens reference alias tokens only |
| Perceptual inconsistency dismissed as "close enough" | Erodes user trust; over time accumulates into a "broken" feel | Enforce via token consumption; design reviews catch token bypass |
